# 평가 프로토콜 — 준비 → 실시 → 결과 정리 → 회고

> 이 패키지가 따르는 평가의 **표준 절차**다. 평가를 즉흥이 아니라 한 번 만든 틀에 걸어
> **일정하게·반복해서·안전하게** 굴린다. 각 단계는 "무엇을 근거로, 무엇을 만들고, 사람이 무엇을 확정하는지"가 정해져 있다.
> ※ 이 문서는 **단원 프로토콜**(`단원-프로토콜.md` — ①해석→②수업 설계→③평가 설계→④운영→결과→회고)의
> **③~회고 구간의 원본**이다. 단원 단위로 진행할 때는 단원-프로토콜.md에서 시작한다.
> 최종 갱신: 2026-07-03

```
준비 ──→ 실시 ──→ 결과 정리 ──→ 회고
설계 일관성   산출물·기록 통일   근거 연결       다시 준비로 ↻
```

모든 단계에 공통인 대원칙 (CLAUDE.md 하드룰):
- **사실은 데이터에서** — 성취기준·평가방식은 `references/`에서 꺼내 쓴다. 없으면 `[확인 필요]`.
- **AI는 초안까지, 확정은 교사** — 모든 생성물은 교사 검토를 전제한 초안이다.
- **비식별** — 학생은 **번호로만** 다룬다. 실명·학번·생년월일은 어떤 파일에도 넣지 않는다.
- **검증** — 산출물을 만들거나 고칠 때마다 `node scripts/lint.mjs` 통과.

## 단계 1 — 준비 (설계 일관성)

**목적**: 공식 근거(성취기준 원문·평가방식) 위에서 평가를 설계한다. 반·해가 바뀌어도 잣대가 흔들리지 않게.
"무엇을 시킬지"부터 막막하면 **구상(브레인스토밍)**에서 시작한다.

| 항목 | 내용 |
|---|---|
| 근거 | `references/교과목록.json` · `references/성취기준/<과목>.json` (별책9 원문 — 성취기준·탐구활동·**핵심아이디어**) · `references/평가기준/<과목>.json` (**KICE 성취수준 A~E 원문** — 루브릭 수준 기술의 근거) · `references/학업성적관리-지침.md` (경남 지침 발췌 — 횟수·40% 규칙·과제형 금지 등 계획 하드룰) · `references/평가유형-카탈로그.md` (유형 10종 + **개념적 렌즈** + **수행지시어 사전**) · `references/수행평가-요소/<과목>-<출판사>.json` (교과서·지도서 추출, 있으면) · `references/수행평가-도구-KICE.json` (도구 카탈로그 — 아이디어 차용용 `참고`) |
| 산출물 | `구상.md`(선택) → `평가계획서.html` · `루브릭.html` + **`루브릭.json`** · `활동지.html` · `meta.json` |
| 스킬 | (`구상` →) `평가계획서` → `루브릭` → `활동지` |
| 교사 확정 | 구상 후보 중 채택·변형, **단원 핵심 아이디어(렌즈)** 확정, 반영비율·평가시기(학교 규정), 배점·수준 기술이 우리 반 수준에 맞는지 |

**개념기반 설계 흐름 (구상 단계에서 권장)**: 개념기반 교육과정과 수업(CBCI)의 흐름을 차용한다 — 성취기준(사실) → **개념적 렌즈** → **단원 핵심 아이디어**(일반화) → 절 분해로 **평가요소** 도출.
"무엇을 시킬지"를 활동이 아니라 **학생이 도달할 한 문장**에서 역방향으로, 이해 중심 교육과정(UbD, 백워드 설계)의 관점에서 설계한다. (근거: CBCI(Erickson, Lanning & French, 2017)·UbD(Wiggins & McTighe, 2005), 국내 적용은 `자료분석/` 연수 자료. 용어 표준: `references/평가유형-카탈로그.md`)

**하드룰**
- 과목 코드를 먼저 확정하고 평가방식(상대/절대_5단계/절대_3단계)에 맞게 분기한다.
- 성취기준은 원문 그대로 인용. 요약·각색 금지.
- **두 종류의 '핵심 아이디어'를 구분한다**: 교육과정 핵심 아이디어(별책9 데이터, 그대로 인용) vs 단원 핵심 아이디어(교사가 렌즈로 만든 문장 = **제안**). 후자를 교육과정 원문인 척하지 않는다.
- 구상의 아이디어는 **근거 라벨**을 단다: `원문(별책9)`(탐구활동·핵심아이디어 원문) / `참고(<출판사> p.N)`(교과서·지도서 요소, 쪽 인용 필수) / `제안`(데이터에 없는 순수 제안 — 제안임을 명시). 라벨을 속이지 않는다.
- 루브릭은 사람용 `루브릭.html`과 **기계용 `루브릭.json`을 쌍으로** 만든다. 수준은 **수행지시어 위계**(설명＞서술＞요약＞나열)로 벌린다.
  이후 채점·lint가 `루브릭.json`의 수준·배점만 근거로 쓴다. (환각은 구조로 막는다)
- 루브릭 수준 기술은 **`references/평가기준/<과목>.json`(KICE)이 있으면 그것을 우선 근거로** 쓴다 — 라벨 `원문(KICE 평가기준)`.
  포털 미제공 성취기준은 지어내지 않고 `제안` 라벨로 교사 확정 전제. (상세: `루브릭` 스킬 2b)
- 반영비율·횟수는 `references/학업성적관리-지침.md`의 규정(수행 환산총점 40%↑, 과제형 금지 등)과 어긋나지 않게. **학교 규정이 최종 우선.**

## 단계 2 — 실시 (산출물·기록 통일)

**목적**: 평가를 치르는 동안 흔들리기 쉬운 두 가지 — 산출물 양식과 관찰 기록 — 을 구조로 잡는다.
학기말에 기억으로 복원하지 말고, **그 순간 '쓸 수 있는 형태'로** 남긴다.

| 항목 | 내용 |
|---|---|
| 근거 | `references/관찰기록-양식.md` (열 정의·비식별 규칙) |
| 산출물 | `수행기록/관찰기록-<반>.csv` — 열: `번호,날짜,관찰한 사실,관련 역량` |
| 스킬 | `관찰기록` — 입력 3종: 즉석 메모 / 수업 녹취록(전사 텍스트) / 수업 녹음 음성(`uv run scripts/transcribe.py`로 전사) → 구조화된 표. 음성 태깅 규약("기록, N번, 사실")은 스킬 문서 참조 |
| 교사 확정 | 기록된 사실이 실제 관찰과 일치하는지 (기록의 원천은 언제나 교사) |

**하드룰**
- 관찰 기록의 학생 식별자는 **번호만**. `번호` 열은 숫자만 허용 (lint 검사).
- '관찰한 사실'은 **행동·발화 등 사실만** 적는다. 해석·평가어("우수함", "성실함")는 결과 단계에서.
- 메모에 없는 내용을 표로 옮길 때 지어내지 않는다. 불명확하면 `[확인 필요]`.
- 수업 녹음을 쓸 경우: **사전 공지·동의**(규정은 학교 지침 `[확인 필요]`) 하에 녹음하고,
  **음성 원본·전사 txt는 Drive 동기화 폴더 밖 로컬 전용**(실명 호명 포함 가능) — 패키지에는 비식별 CSV만 들어온다.
  녹취록의 화자 미상 발화를 특정 학생에게 **추정 배정 금지** — 번호 확정은 교사만 한다.

## 단계 3 — 결과 정리 (근거 연결)

**목적**: 채점과 과세특을 '증거'에 연결한다. **관찰 기록 → 루브릭 채점 → 점수/과세특 초안**.
이의신청이 와도 루브릭 단계·관찰 근거로 차분히 설명할 수 있게.

| 항목 | 내용 |
|---|---|
| 근거 | `수행기록/관찰기록-<반>.csv` + `루브릭.json` (이 둘 **밖의 근거로 채점 금지**) |
| 산출물 | `산출물/채점표-<반>.csv` · `산출물/과세특-초안-<반>.md` |
| 스킬 | `채점표` → `과세특연계` |
| 교사 확정 | 모든 점수·과세특 문장. 특히 `[확인 필요]` 행은 교사가 직접 채운다 |

**하드룰**
- 수준·배점은 `루브릭.json`에 실재하는 값만 쓴다 (lint가 대조).
- 모든 채점 행에 **근거(관찰 날짜·사실 요약)를 남긴다**. 근거 없는 점수 금지.
- 관찰 기록이 부족한 학생·요소는 점수를 지어내지 않고 `[확인 필요]`로 둔다.
- 과세특은 '**사실 → 역량**' 변환: 관찰된 사실 문장에서 출발해 역량 서술로 잇는다.
  기재요령(`references/과세특-기재요령.md`)의 금지표현·분량 준수 (lint 검사).

## 단계 4 — 회고 (다시 준비로)

**목적**: 이번 평가에서 흔들린 것을 기록해 **다음 준비 단계의 입력**으로 만든다. 순환이 여기서 닫힌다.

| 항목 | 내용 |
|---|---|
| 근거 | `산출물/채점표-*.csv` (수준 분포·[확인 필요] 빈도) + 교사 소감 |
| 산출물 | `회고.md` — 잘 작동한 것 / 흔들린 것 / 다음에 고칠 것(대상 파일 명시) |
| 스킬 | `회고` |
| 교사 확정 | 개선 항목의 채택 여부 — 회고가 확정되면 다음 학기 루브릭·계획서·CLAUDE.md에 반영 |

**하드룰**
- 개선 항목은 반드시 **어느 파일을 고칠지**(루브릭.json 배점? 관찰기록 양식? 활동지 문항?)까지 적는다.
- 회고 없이 다음 학기 산출물을 복사·재사용하지 않는다.

```
assessments/<과목>-<과제명>/
├── meta.json                      과목·성취기준·반영비율 (lint 검증)
├── 구상.md                        [준비] 브레인스토밍 후보안 (선택)
├── 평가계획서.html                 [준비]
├── 루브릭.html · 루브릭.json       [준비] 사람용 · 기계용 쌍
├── 활동지.html                    [준비]
├── 수행기록/
│   └── 관찰기록-<반>.csv           [실시] 번호,날짜,관찰한 사실,관련 역량
├── 산출물/
│   ├── 채점표-<반>.csv             [결과] 근거 열 포함
│   └── 과세특-초안-<반>.md         [결과] 사실 → 역량
└── 회고.md                        [회고]
```

## 단계별 책임 요약 — "초안까지가 AI, 확정은 교사"

| 단계 | AI(패키지)가 하는 것 | 교사가 확정하는 것 |
|---|---|---|
| 준비 | 아이디어 후보(근거 라벨), 성취기준 원문 인용, 계획서·루브릭·활동지 초안 | 아이디어 채택·변형, 배점·수준 기술·반영비율·시기 |
| 실시 | 즉석 메모의 구조화(표 변환) | 기록된 사실의 진위 (원천은 교사) |
| 결과 | 기록×루브릭 대조 채점 초안, 과세특 초안 | 모든 점수와 문장, `[확인 필요]` 해소 |
| 회고 | 채점표 통계 요약, 개선 후보 정리 | 개선 항목 채택, 다음 학기 반영 |
